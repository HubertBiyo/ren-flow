# ren-inbox · 讨论线与文件格式参考

> 何时读这份:实际写入 / 重排 inbox 文件、或一条灵感要来回拍板需要挂讨论线时。SKILL.md 的流程与状态标已够日常 capture;这份给细排版与示例。

## 讨论线（要来回拍板的条目才挂）

一条灵感一句话定不了时,在它下面挂 2 空格缩进的子 bullet 记来回,用短前缀 `我：` / `答：` 标谁说的(不用表情),方便以后直接接着聊:

```markdown
- ⬜待确认 14:15 [需求] 同步用户风控信息接口，与提现同源落 user_risk_record
  - 我：走 sync 还是单独接口
  - 答：建议 sync 附带，省客户端一次往返
```

- 一句一行,别把整段塞成一长行;话长就拆成几行短句。
- 主行行首的 `⬜` 状态标**始终在主行**,讨论线再长都不影响 `grep ⬜`。
- 球在谁手里就翻子态:抛出结论等你点头 → `⬜待确认`;你答完抛回去 → `⬜待消化`。
- 闭环就坍缩:翻 `✅/❌` 时把讨论线收成主行的去向括注,别留一堆草稿。

## 布局：跟 ren-log 同一套观感

排版参考 `ren-log`:`##` 分区 + 短 bullet + 条目间空一行,扫起来有呼吸感。两种模式各自的形态:

- **capture**:按时间往当天文件末尾 append,不分组 —— 随手记不该停下来想归哪类。每条主行一句话,上下文挂 detail 子行,别让任何一行拖很长。
- **triage**:条目攒多了,整理时把当天文件重排成 `## 类型` 分区(`## 需求` / `## 优化` / `## 灵感` / `## bug` / `## 调研`),等你拍板的提一个 `## ⬜待确认（需你拍板）` 放最上面。**可选整理,不强制**;条目少就保持扁平时间序,别为分组而分组。

不管哪种,**每条都短**:主行一句话,要补充就换行成几条短 detail,不要一行写成一大段(那是现在最伤观感的地方)。

## 文件格式

capture 态(扁平时间序):

```markdown
# 2026-06-06 (周六)

- ⬜待消化 11:25 [优化] 提现风控命中后给运营主动告警，不只落库

- ❌弃 14:30 [需求] 补餐要不要服务端校验真看广告  (信任客户端)
  - 我：当前不做对吧
  - 答：对，信任客户端；广告 SSV 列后续增强
```

triage 后(按类型分区,参考 ren-log):

```markdown
# 2026-06-05 (周五)

## ⬜待确认（需你拍板）

- ⬜待确认 18:52 [需求] 第三方账号未绑定时标记成功 + 落真实错误，需先定口径
  - 我：这个做了没
  - 答：没做，现状相反；要做先定口径……

## 需求

- ✅已转 11:29 [需求] 新人引导倒计时 + 欢迎奖励并进活动面板  (74b5bac)

## 优化

- ✅已转 17:04 [优化] 用户列表支持按渠道筛选 + 展示  (全链路打通)
```

格式要点:

- 标题 `# {YYYY-MM-DD} ({周几})`
- 主行 `- {状态标} HH:MM [类型] 一句话标题  (去向/原因)`;状态标行首、去向放行尾括号;补充挂缩进 detail / 讨论子行,一行别拖长
- 条目之间空一行,留白
- 类型是「种类」不是业务域:`[需求] [灵感] [优化] [bug] [调研]`;域信息写进正文,别新造 `[风控]` / `[订单]` 这种域标
- **纯文字,不用 markdown 加粗,不用表情**
