# 数据访问与资源红线清单

> **何时读**:实现步骤涉及 DB 查询 / 缓存 / 循环调外部资源(DB、Redis、HTTP)时,**动手前**过一遍;该步写完后拿它对照 diff 自查。
> 这类问题的共性:写的当下不拦,就要等评审甚至生产慢查询才暴露 —— 属于返工成本最高的一类。
> 各栈具体写法(driver API / LINQ / ORM 细节)以项目 attention.md 与栈规范技能为准;本清单只定红线,栈无关。

## 查询

- **按需取字段**:业务只用少数字段时用投影(projection),不整文档 / 整行捞回来。
- **能带主键 / 唯一键就带上**:已知 id 的查询把 id 放进条件,别只靠二级条件扫。
- **索引命中**:新查询的过滤 / 排序字段对照项目索引清单;大表查询不允许无索引扫描。
- **必须有界**:列表查询必须分页或带上限;禁止恒真条件(`where(true)` / `1=1` 式)兜底 —— 容易演变成全表拉取甚至死循环。
- **条件不重复**:同一查询里不堆叠语义重复的条件;新查询与同语义旧查询的过滤口径对齐(软删、状态、应用隔离字段)。

## 重复访问

- **拿过的对象不再查**:方法调用链里上游已查出的对象,往下**传参**,不要下游「为了方便」再查一遍。
- **一次请求内不重复**:同一方法 / 同一请求内不多次打开连接、不重复查同一条数据。

## 循环与批量

- **禁止循环内逐条调 DB / Redis / 外部接口**(N+1):先收集 key,再用批量方式(IN 查询 / MGET / 批量接口)一次取回,内存建映射后分发。
- **要多条就用批量方法**:不要循环调「查单条」凑多条;批量方法缺失时优先补批量方法,而不是迁就单条接口。

## 缓存

- **按筛选条件 + 分页粒度缓存**:用户不会看全量,不要一次查出 / 缓存所有数据再内存翻页。
- **穿透防护**:查无结果也写入短 TTL 的空值缓存,防止穿透打 DB。
- **只缓存用到的字段**:不把整个对象塞进缓存占内存。
- **key 简洁分层**:`{项目}:{模块}:{实体}:{id}` 式统一分层,不拼无结构长串。

## 异常与日志

- **catch 不许吞异常**:捕获后至少记一条带上下文(关键参数、id)的日志,或重新抛出;静默 catch 是排障黑洞。
- **不打非必要调试日志**:热路径上克制 info / debug 输出;留下的每条日志要能回答一个排障问题。

## 并发与异步

- **异步全链路一致**:不 fire-and-forget(调了 async 不等待);不在异步链路里同步阻塞(`.Result` / `.Wait()` 式)。

## 集合与内存

- **列表转字典看 key**:确认 key 字段类型一致且唯一(重复 key 直接抛异常)。
- **同一集合不多次遍历**:按不同条件取数时,一次遍历完成分组,或先转字典 / 查找表。
- **惰性序列先物化**:`IEnumerable` / generator 会被多次消费时先物化(`.ToList()` 式),避免底层查询重复执行。

## 事务

- **事务要短**:纯查询移到事务外;事务内只保留必须原子化的写操作。

## 其他

- **固定值用枚举 / 常量**:魔法数字、魔法字符串收敛成枚举或常量,改一处全生效。

## 设计期条款(spec 阶段就该定,不留到写代码时即兴)

缓存策略(缓存什么、什么粒度、TTL)、索引方案、批量接口形态 —— 属于方案设计,`ren-spec` 阶段对齐;build 阶段发现缺口回 spec 补,不现场发明。
