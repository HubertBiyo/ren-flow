# ren-spec · PRD 模式参考

> 何时读这份:跨角色 / 跨团队需求(产品 / 客户端 / 后端三方对齐、SLA / 接口契约要冻结)时。纯个人 / 单栈需求不用读 —— 回 SKILL.md 走快路 / 全程即可。

需要正式 PRD 时,用下面这套结构产出 `prd.md`,与 spec 文档并存。**单一事实来源**:避免「微信群一句 + 文档一句」双轨;冲突以 PRD 更新为准。

```
1. 执行口径       —— 唯一有效说明是哪份；原始口述仅溯源
2. 背景与问题 / 目标
3. 成功标准（可验证）—— 运营、接口、时效、隔离等可判定条目
4. 范围 + 本期非目标 —— 非目标单独成块（Shape Up 式「明确不做」）
5. 术语表          —— 跨角色对齐 `BookSKUId` / 客户端名 / 名单含义
6. 风险与合规      —— 2-5 行：误操作、审计、隐私
7. 功能需求 + 接口清单表（路径/方法/备注）
8. 接口契约（冻结表）—— 字段名/类型/语义/true|false 含义/默认值/示例；与 OAS 二选一或互链
9. 实现建议（可选）—— 缓存、SLA；标注**不替代产品 SLA**
10. 扩展性与本期边界 —— 长期模型 vs 本期交付；**一次写清是否预留字段**
11. 开发前需确定 —— 责任方、状态列、**最小可开工条件**
12. 验收 —— 要点列表 + **Given/When/Then** 场景表
```

写作硬约束:**布尔与枚举必有文字语义**（禁止只写「加一个 bool」不写 true 表示什么）;**验收要能翻译成测试用例或手工步骤**;SLA / 字段名未定处标「待定」并列入开发前清单。

> 起草模板与纪律(1~4 节结构、起草纪律、踩坑清单)同 `full-spec.md`,PRD 与 spec 文档并存时两份都按各自结构写。
