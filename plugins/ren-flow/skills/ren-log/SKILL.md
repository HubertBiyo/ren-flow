# ren-log

## 启动必读

先 `Read .ren-flow/attention.md`。

**工作区模式下 journal 不分域** —— 日志是你一个人的流水,始终落在工作区根 `.ren-flow/journal/`,不进 `{domain}/`。

## 这个技能干什么

把工作进度记进每日 journal,方便回顾、写日报、做迭代汇总。两种模式:

- **capture(默认)** —— 把当前对话窗口干的活汇总成一条带时间戳的 ≤100 字 bullet。随手记、多窗口安全。触发:`/ren-log`、「同步日志」「记一下今天」「今日总结」。
- **rollup(整理)** —— 把当天所有 capture bullet 综合成结构化日报(今日重点 / 任务清单 / 备忘)。一天结束时跑一次。触发:`/ren-log 整理`、「整理今天」「出日报」「写日报」。

两层关系:capture 是全天的原始捕获(每窗口一条,绝不互相覆盖);rollup 是一天一次的综合(把碎 bullet 重组成可汇报的日报)。capture 轻、rollup 重,各司其职。**它不取代项目已有的重型日志系统** —— 见「落点判定」。

产物:`.ren-flow/journal/{YYYY-MM}/{YYYY-MM-DD}.md` —— 月文件夹 + 一天一文件。

## 落点判定（两种模式都先做）

读 `.ren-flow/attention.md` 的 `journal:` 配置:

- **`journal: external`** —— 项目日志由别的系统维护。ren-log **让位**:不写,告诉用户「本项目日志由原系统维护,请用对应技能」,本技能结束。
- **`journal: .ren-flow` 或未配置** —— 用 `.ren-flow/journal/`,继续。
- **未配置,但 `Glob` 发现项目已有日志目录**(如 `docs/**/journal/`)—— 停下问用户一次:「检测到 `X` 像是已有日志系统,ren-log 让位还是用 `.ren-flow/journal/`?」结果写进 attention.md,以后不再问。

## capture 模式流程

### 1. 先读今天的文件（关键,不能跳）

`Read .ren-flow/journal/{YYYY-MM}/{YYYY-MM-DD}.md`:

- **已存在** → 待会儿追加 bullet,不覆盖别的窗口已写的内容
- **不存在** → 待会儿新建(含 `# {YYYY-MM-DD} ({周几})` 标题);月文件夹不存在则一并建

### 2. 取时间标识

跑 `date "+%H:%M"` 取当前时分作为本条记录标识 —— 区分同一天不同窗口写的记录。

### 3. 汇总本窗口的功能点

收集本对话窗口今天做的事(对话上下文 + 当天 git commit + `.ren-flow/` 当天新增制品),砍掉过程废话,只留迭代功能点,**≤100 字**,多个点用分号隔开。

### 4. 写入

bullet 落在文件的「原始记录」区:

- 文件已 rollup 过(有 `## 🕐 原始记录` 小节)→ 追加到该小节末尾
- 未 rollup → 直接追加在标题下

写法 `- HH:MM {汇总}`,按时间先后排。

- 文件里**没有**雷同 bullet → 追加一条
- **有**雷同(同窗口之前记过、这次是补充)→ 就地更新那条,不新增

### 5. 给用户看一眼写入的内容

## rollup 模式流程

### 1. 落点判定 + 读今天的文件 + 收集素材

- `Read` 今天的 journal,取出所有 `- HH:MM` 原始 bullet
- 跑当天各相关仓库 `git log` 取 commit
- 扫 `.ren-flow/` 当天新增 / 改动的制品(specs / fixes / notes)

### 2. 综合成结构化日报

把碎 bullet + commit + 制品重组成三节(格式见下「rollup 日报格式」):

- **今日重点**:3-6 条,粒度「一块大活一条」。**按产品 + 端聚合** —— 同一产品同一端(后台 / 移动端 / 服务端 / H5)的多个模块并成一条,写成 `{产品} {端} {模块A、模块B、模块C}调整/完善/开发`;朴实动词收尾,**不展开实现细节**(细节进任务清单)。跨窗口同类合并。
  - **只放功能 / 迭代,不放迭代内的修复**:联调时发现的 bug 修复、报错排查、状态修正等属于「迭代内的修复」,**进任务清单不进今日重点**。今日重点回答「今天推进了什么功能 / 迭代」,不是「今天修了什么 bug」。(例:`欢迎奖励领取联调修复:金额报错、卡死重试` 这类不进重点,进任务清单的对应迭代分组下。)
  - 例:`后台 规则配置完善、国家配置调整、用户管理开发` / `移动端 用户信息同步、新人奖励活动开发`
  - **大白话铁律(今日重点 + 任务清单都适用)**:日报会贴给非本项目同事 / 上级看。**禁止内部里程碑代号(M1/M3/M5、Phase4)、技术术语(Decimal128 / MVP / lazy-init / OpenAPI)、内部字段名(如 `base_value` 这类库字段)**。从 commit message / roadmap 搬词时必须翻译成业务大白话(如 `M5 风控钩子`→`风控:命中转人工`,`资金口径改 Decimal128`→`金额改成填多少就是多少`)。判据:一个不懂这个项目的人能不能看懂。
- **任务清单**:按迭代 / 功能分组(`###` 子标题),每组下列具体任务(同样守大白话铁律)
- **备忘**:待办、待决定、会议要点

不臆造内容 —— 只综合 bullet / commit / 制品里真实有的;拿不准的列进备忘而非重点。

### 3. 写回文件

文件重写为:`# 标题` + 三节日报 + `## 🕐 原始记录`(**保留全部 `- HH:MM` bullet** 作审计线索,不删)。

rollup **幂等** —— 当天再次跑(又新增了 bullet)就把原始记录重新综合一遍,覆盖旧的三节。

### 4. 给用户看一眼日报

## journal 文件格式

实际写入 capture bullet 或 rollup 日报、要确认排版与示例时,`Read references/format.md` —— capture 期 / rollup 后的完整示例与格式要点都在那里。关键约束就两条,常驻记住:capture 是 `- HH:MM {≤100 字汇总}`;rollup 三节固定 emoji 标题(`🎯 今日重点` / `📝 任务清单` / `💡 备忘`)+ 保留 `## 🕐 原始记录`;**全程纯文字不加粗**。

## 退出条件

- [ ] 已做落点判定;`journal: external` 时已让位,未误写
- [ ] capture:本窗口记录是一条 `- HH:MM` bullet,≤100 字,纯文字无加粗,雷同已就地合并
- [ ] rollup:三节齐全,原始 bullet 完整保留在 `## 🕐 原始记录`,跨窗口同类已合并
- [ ] 记录写进 `journal/{YYYY-MM}/{YYYY-MM-DD}.md`,月文件夹已建

## 容易踩的坑

- 项目已有重型日志系统还硬记 ren-log —— 检测到就让位,别造第二套
- **capture 没先读今天的文件就写** → 覆盖掉别的窗口已写的内容
- capture 把别的窗口已记的内容再抄一遍 —— 雷同应合并,不重复
- capture 写成流水账 / 超 100 字 / 用 markdown 加粗
- **rollup 把原始 bullet 删掉** —— 必须完整保留在 `## 🕐 原始记录` 作审计线索
- rollup 臆造 bullet / commit 里没有的内容 —— 只综合真实素材
- rollup 当成每窗口都跑 —— 它是一天一次的综合,频繁跑是浪费
- **rollup 把 commit / roadmap 里的内部代号、技术术语直接搬进今日重点 / 任务清单** —— M3/M5/Phase4、Decimal128/MVP/lazy-init、内部字段名都要翻译成业务大白话,日报是给非本项目同事看的
