# ren-flow 路由 eval

给 ren-flow 技能集装的一层**回归安全网**。改了任何技能的 `description:` 或 `ren` 路由表后,重放一遍语料,确认「用户这么说 → 该去哪个技能」没有变差。

## 为什么需要

ren-flow 有 16 个 ren-* 技能,靠两样东西决定一句话路由到谁:

1. `ren` 路由器里的路由表(`plugins/ren-flow/skills/free/SKILL.md`)
2. 每个技能 frontmatter 的 `description:`(触发词写在这里)

这两处任何一处一改,路由就可能悄悄退化——某句话本该去 A 却被 B 抢走,或落进澄清。靠手感迭代 16 个技能时,这种退化没人接得住(参见 memory `feedback_review_vs_simplify_division` 里 review 三件套撞车的教训)。这层 eval 把「路由对不对」从手感变成**可重放的判定**。

## 怎么跑(重放流程)

环境里没有可调 API 的自动跑测框架,所以这是一套**给 Claude 执行的冻结输入重放**,不是脚本。让 Claude 按下面做:

1. **冻结路由面**:读 `plugins/ren-flow/skills/free/SKILL.md` 的路由表 + 所有 `ren-*/SKILL.md` 的 `description:` 行。**只用这两样**作判断依据——不许读技能正文、不许用语料里的「期望/考点」列(那是答案,判完才比对)。
2. **逐条盲判**:对 `routing-cases.md` 每条「用户原话」,只凭第 1 步的路由面,预测它会路由到哪个技能(或澄清 / 让位 / 直接改)。
3. **比对出表**:预测 vs 期望,出一张结果表 `# | 原话 | 期望 | 预测 | ✓/✗`。
4. **诊断每个 ✗**:判断偏差属于哪类——
   - **路由面缺陷**:description 触发词缺失 / 重叠、路由表某行含糊 → 改路由面(description 或路由表),这是 eval 抓到的真回归。
   - **语料问题**:这句话本就该澄清 / 确实两可 / 期望写错 → 改 `routing-cases.md`。
   - 区分原则:一个没读过技能正文的人,光看 description 能不能判对。判不对 = 路由面缺陷,不是模型笨。
5. **收敛**:改完路由面后重放,直到全绿或剩下的 ✗ 都是「有意的两可,期望已是澄清」。

## 什么时候必须跑

- 改了任何 `ren-*` 的 `description:`
- 改了 `ren` 路由表
- 新增 / 删除 / 合并 ren-* 技能
- 怀疑两个技能触发词撞车时(先加一条能区分二者的语料,再重放)

> 渐进披露改造(把技能正文拆进 `references/`)**不动 description 与路由表**,理论上不影响路由——但拆完跑一遍是廉价的确认。

## 维护语料

- 每次真踩到「路由跑偏」的实例,补一条进 `routing-cases.md`(像 bug 回归测试一样,踩一个补一个)。
- 撞车类务必成对收(J 组、F 组那样),单条测不出「该去 A 没被 B 抢」。
- 语料只测 ren-* 内部路由;跨体系(review-heavy / code-review / simplify 等非 ren-* 技能)不收,那是 memory 管的范围。
