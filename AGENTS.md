# ren-flow 技能维护规范

## 核心原则

技能的唯一价值 = **补偿模型局限** + **编码项目/流程专有知识**。模型自己能做好的事不写进技能 —— 那是熵。

## 硬约束

- 单个 `SKILL.md` 不超过 **300 行**，超了必须拆到同目录 `reference.md`。
- 技能之间**不相互耦合**：A 技能运行时只能看到自己包内的文件，不能引用 `B-skill/xxx.md`。跨技能要共享的口径，走项目 `.ren-flow/` 这一层（由 `ren-init` 释放），子技能用项目相对路径读。
- 增加 / 修改技能时，同步更新其他技能里相关的表述（尤其 `ren` 路由表、`README.md` 技能表）。
- 正文中文为主；代码、字段名、命令、API 路径保持英文原文。
- 言简意赅。每句话要么改变读者的判断，要么改变读者的下一步动作。

## frontmatter

每个 `SKILL.md` 必含：

```yaml
---
name: ren-xxx
description: 一句话职责 + 触发场景（含「触发：」关键词，供路由匹配）
---
```

## 何时新增技能

以下**全部**满足才值得新建：

1. 模型在没有技能时反复犯同一类错误
2. 该错误无法靠更新 `attention.md` 解决（属于工作流程，不是规范条目）
3. 触发场景足够具体，不与现有技能歧义

当前技能数 16（含 ren-inbox）。每加一个先问：能不能并进现有的？再增（≥17）触发一次全面 review。

## 何时删 / 并技能

- 两个技能触发词高度重叠 → 合并，留更具体的
- 技能内容已被项目 `attention.md` 完全覆盖 → 删
- 模型升级后已能自行处理 → 删

## SKILL.md 推荐结构

启动必读 → 职责说明 → 流程 / 路由 → 退出条件 checklist → 容易踩的坑 → 相关文档。
